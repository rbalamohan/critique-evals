
  ---
  Benchmark Study: Does Model Diversity Improve Code Review Quality?

  ---
  Background & Motivation

  A common pattern in AI-assisted software development is to use one model to generate code and another to review it. The open question is: does using different models for these two roles produce better outcomes than using the same model
  for both?

  This study was designed to answer that question empirically, using SQL code generation and review as the test domain.

  ---
  The Setup

  Domain: A logistics operations database (Snowflake SQL) representing task management — organizations, workers, and task completion records.

  Models tested:
  - Coder: Claude (Sonnet 4.6) or GPT (GPT-5.4)
  - Critic: Claude (Sonnet 4.6) or GPT (GPT-5.4)
  - All 4 combinations tested: Claude→Claude, Claude→GPT, GPT→Claude, GPT→GPT

  Three SQL tasks of increasing complexity:

  ┌───────────────────────┬─────────────────────────────────────────────────────────────────────────────────┐
  │         Task          │                                  What it tests                                  │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
  │ Weekly Success Report │ Basic aggregation — weekly task volume and success rate for an org over 8 weeks │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
  │ Failure Breakdown     │ NULL handling + window functions — distribution of failure reasons with % share │
  ├───────────────────────┼─────────────────────────────────────────────────────────────────────────────────┤
  │ Daily Volume Trend    │ CTE + LAG window function — day-over-day task volume change over 14 days        │
  └───────────────────────┴─────────────────────────────────────────────────────────────────────────────────┘

  Scale: 20 independent coder runs per task × 3 tasks = 60 SQL queries generated per model, each reviewed by both critics independently.

  ---
  How We Measured Quality

  Phase 1 — Code Generation: Each coder model generated SQL independently 20 times per task. We measured how consistent the outputs were across runs.

  Phase 2 — Bug Detection: Before sending to the critic, we injected a deliberate bug into each generated SQL (swapping a correct join condition for a wrong one). We then measured how often each critic correctly flagged the corrupted SQL
  as flawed.

  Phase 3 — False Positive Check: We also gave each critic a verified-correct ground truth SQL (written by a domain expert). A good critic should accept it — rejecting correct code is a false positive.

  ---
  Key Observations

  1. Claude is a substantially better critic than GPT.
  Averaged across all tasks, Claude detected 52–60% of injected bugs versus GPT's 30–35%. This is a consistent ~2× gap across every task and every coder combination.

  2. Critic choice matters more than coder choice.
  When GPT writes code and GPT reviews it (same-model pair), only 35% of bugs are caught. Replacing the critic with Claude — while keeping GPT as the coder — raises detection to 30–40% on identical code. The reviewer, not the writer,
  determines quality gate effectiveness.

  3. Claude generates highly stable, convergent SQL; GPT generates diverse, variable SQL.
  On the Failure Breakdown task, some GPT outputs across different runs were only 6% similar to each other — essentially different queries solving the same problem differently. Claude produced near-identical SQL every time. This has
  implications for test coverage: a same-model Claude→Claude pair always evaluates the same narrow output, while a GPT coder naturally stress-tests the critic with structural variation.

  4. No false positives from either critic.
  Both models correctly accepted all verified-correct SQL. The detection gap is therefore not an artifact of GPT being more lenient — GPT simply misses real bugs that Claude catches.

  ---
  Why This Supports the Paper's Argument

  This study provides empirical evidence for two distinct failure modes in same-model code review pipelines:

  Failure Mode 1 — Blind spot bias. A model reviewing its own style of output may reconstruct intent rather than evaluate correctness. GPT reviewing GPT-generated code caught only 35–50% of bugs; Claude reviewing the same GPT-generated
  code caught 30–40% more. A different model brings an independent perspective.

  Failure Mode 2 — Convergence collapse. A same-model coder+critic pair like Claude→Claude will always evaluate the same narrow SQL structure. Claude's 95–97% output consistency means the critic is never exposed to alternative
  interpretations. GPT's 44–89% consistency means a cross-model pair naturally evaluates a wider range of valid approaches, surfacing gaps a stable same-model pair would never encounter.

  Together these argue: using the same model for both generation and review creates a system that is self-consistent but not self-correcting. Cross-model pairing breaks this by introducing independent judgment (different critic) and
  structural variation (different coder).
